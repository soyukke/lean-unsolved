# 形式化フェーズ — 発見を恒久的な成果に変換

## 対象: $ARGUMENTS

探索サイクルで得られた発見をLean 4形式証明に変換し、プロジェクトの知識ベースに定着させる。
「主予想が解けなくても、副産物の形式化は確かな成果」という原則に基づく。

## 手順

### 1. 形式化候補の特定

以下を読み、形式化すべき対象を決定する:

1. `Unsolved/explorations/$ARGUMENTS/SUMMARY.md` → 「形式化候補」セクション
2. `Unsolved/explorations/queue.json` → `formalization_value` が「定理候補」「補題候補」「副産物候補」のdoneアイテム
3. 直近の探索結果ファイル（`exploration_NNN.md`）→ `formalization_value` と `lean_relevant` フィールド

**優先順位**:
1. **定理候補** — 独立した数学的主張で sorry なし証明が見込めるもの
2. **副産物候補** — 主予想と無関係だが Index.lean に追加する価値があるもの
3. **補題候補** — 既存モジュールの自然な拡張

### 2. 形式化対象の分析

選択した候補について:

- **数学的主張の明確化**: 自然言語で定理を正確に述べる
- **関連Leanモジュールの読み込み**: `lean_relevant` で指定されたモジュールを Read で読む
- **既存定理との関係**: 同じ結果が既に形式化されていないか Grep で確認
- **証明戦略**: どの tactic で証明するか方針を立てる
  - `omega`: 線形算術
  - `decide` / `native_decide`: 有限ケース検証
  - `simp + split`: 場合分け
  - `induction`: 帰納法
  - `norm_num`: 数値計算

### 3. Lean証明の作成

**配置先の決定**:

| 形式化の種類 | 配置先 |
|------------|--------|
| 既存モジュールの拡張 | そのモジュール末尾に追加 |
| 新しい独立した結果 | 最も関連の深いモジュールに追加 |
| 完全に新しいトピック | 新ファイル作成（バレルに追加） |

**証明作成のルール**:
- **sorry は絶対に使わない**。証明が完成しない場合は中止し、理由を記録する
- docstring を付ける（定理の意味を日本語で説明）
- 既存の命名規則に従う（`collatzReaches_*`, `stoppingTime_*` 等）
- `set_option maxHeartbeats` が必要な場合はコメントで理由を記載

**証明作成の手順**:
1. Edit ツールでモジュールに定理を追加
2. `lake build [モジュール名]` でビルド確認
3. エラーがあれば修正（最大3回まで試行）
4. 3回失敗したら中止し、「形式化未完了」として記録

### 4. ビルド検証

```bash
/Users/soyukke/.elan/toolchains/leanprover--lean4---v4.29.0-rc4/bin/lake build [モジュール名]
```

- warning は許容（ただし `sorry` 関連は不可）
- error は必ず修正

### 5. Index.lean の更新

形式化が成功したら、`Unsolved/Index.lean` を更新:

- 該当するセクションに新しい定理を追加
- 再利用性の評価（★/★★/★★★）を記載
- どのファイルのどの定理かを明記

### 6. 知識ベースの更新

- **SUMMARY.md**: 「形式化状況」セクションを更新（sorry なし件数の増加）
- **INDEX.md**: 形式化が完了した探索のエントリに「Lean形式化済み」のマークを追加
- **queue.json**: 該当アイテムに `"formalized": true` を追加

### 7. 結果報告

```
## 形式化結果
- 対象: [定理名/概要]
- 配置先: [Lean モジュール名]
- ステータス: 完了 / 未完了（理由: ...）
- sorry: なし
- Index.lean: 更新済み / 未更新

## 追加された定理
- `theorem_name`: [定理の内容を1行で]

## 次のアクション
- [キューに形式化候補がまだある場合] → `/formalize` を継続
- [候補を使い切った場合] → `/brainstorm` または `/explore-batch` に戻る
```

## 重要な原則

- **sorry ゼロ**: 形式化の価値は sorry がないことで担保される。中途半端なら出さない
- **小さく確実に**: 1回の形式化で大量の定理を追加するより、1-2個を確実に証明する
- **副産物を大切に**: 主予想への直接的貢献がなくても、独立した定理は成果
- **ビルド必須**: `lake build` が通らない変更はコミットしない
- **差分を意識**: 既存定理の些細な言い換えではなく、本当に新しい結果を形式化する
